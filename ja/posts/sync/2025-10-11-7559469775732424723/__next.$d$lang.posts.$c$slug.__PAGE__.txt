<h3 id="1-stdio-standard-inputoutput">1. Stdio (Standard
Input/Output)</h3>
<p><strong>协议本质：</strong></p>
<ul>
<li>基于 <strong>进程间通信（IPC）</strong>  的经典模式</li>
<li>利用 Unix/Linux 的管道机制（pipe）</li>
<li>消息格式：JSON-RPC 2.0 over newline-delimited JSON</li>
</ul>
<p><strong>工作流程：</strong></p>
<pre class="mermaid"><code>sequenceDiagram
Client-&gt;&gt;MCP Server (Child Process): spawn(command, args)
Client--&gt;&gt;MCP Server (Child Process): stdin: {&quot;method&quot;:&quot;init&quot;}
MCP Server (Child Process)--&gt;&gt;Client: stdout: {&quot;result&quot;:{...}}
Client--&gt;&gt;MCP Server (Child Process): stdin: {&quot;method&quot;:&quot;tools&quot;}
MCP Server (Child Process)--&gt;&gt;Client: stdout: {&quot;result&quot;:{...}}
Client-&gt;&gt;MCP Server (Child Process): kill()</code></pre>
<p><strong>使用场景：</strong></p>
<ul>
<li>✅
本地命令行工具（如 <code>npx</code>, <code>python</code>, <code>node</code>）</li>
<li>✅ 开发者工具（LSP、formatter、linter）</li>
<li>✅ 离线场景（无网络依赖）</li>
<li>✅ 需要环境变量隔离的工具</li>
</ul>
<p><strong>优点：</strong></p>
<ol type="1">
<li><strong>零网络开销</strong>：直接进程通信，延迟 &lt; 1ms</li>
<li><strong>安全性高</strong>：不暴露端口，不受网络攻击</li>
<li><strong>简单直接</strong>：spawn 一个进程就能用</li>
<li><strong>资源控制精确</strong>：可以随时 kill 进程</li>
</ol>
<p><strong>缺点：</strong></p>
<ol type="1">
<li><strong>只能本地</strong>：无法跨机器调用</li>
<li><strong>进程启动开销</strong>：每次调用可能需要启动新进程（如 npx
需要下载依赖）</li>
<li><strong>无法共享实例</strong>：每个客户端都启动独立进程，浪费资源</li>
<li><strong>调试困难</strong>：错误信息只能从 stderr 获取</li>
</ol>
<p><strong>典型实现：</strong></p>
<ul>
<li>Claude Desktop 的本地 MCP 工具</li>
<li>VS Code Extension 的 Language Server Protocol (LSP)</li>
<li>Git 的 <code>filter-branch</code> 等管道命令</li>
</ul>
<hr />
<h3 id="2-sse-server-sent-events">2. SSE (Server-Sent Events)</h3>
<p><strong>协议本质：</strong></p>
<ul>
<li>基于 HTTP 的 <strong>单向推送</strong> 协议</li>
<li>服务器主动向客户端发送事件流</li>
<li>底层是 HTTP 长连接（Content-Type: text/event-stream）</li>
</ul>
<p><strong>工作流程：</strong></p>
<pre class="mermaid"><code>sequenceDiagram
Client-&gt;&gt;MCP Server (HTTP): GET /events (Accept: SSE)
MCP Server (HTTP)--&gt;&gt;Client: 200 OK (keep-alive）
MCP Server (HTTP)--&gt;&gt;Client: event: tool_update，data: {...} 
MCP Server (HTTP)--&gt;&gt;Client: event: notification，data: {...} 
Client-&gt;&gt;MCP Server (HTTP): (connection kept open)</code></pre>
<p><strong>消息格式：</strong></p>
<div class="sourceCode" id="cb3"><pre class="sourceCode js"><code class="sourceCode javascript"><span id="cb3-1"><a href="#cb3-1" aria-hidden="true" tabindex="-1"></a><span class="bu">event</span><span class="op">:</span> message</span>
<span id="cb3-2"><a href="#cb3-2" aria-hidden="true" tabindex="-1"></a>data<span class="op">:</span> {<span class="st">&quot;type&quot;</span><span class="op">:</span><span class="st">&quot;tool_update&quot;</span><span class="op">,</span><span class="st">&quot;payload&quot;</span><span class="op">:</span>{<span class="op">...</span>}}</span>
<span id="cb3-3"><a href="#cb3-3" aria-hidden="true" tabindex="-1"></a>id<span class="op">:</span> <span class="dv">12345</span></span>
<span id="cb3-4"><a href="#cb3-4" aria-hidden="true" tabindex="-1"></a></span>
<span id="cb3-5"><a href="#cb3-5" aria-hidden="true" tabindex="-1"></a><span class="bu">event</span><span class="op">:</span> notification</span>
<span id="cb3-6"><a href="#cb3-6" aria-hidden="true" tabindex="-1"></a>data<span class="op">:</span> {<span class="st">&quot;message&quot;</span><span class="op">:</span><span class="st">&quot;Task completed&quot;</span>}</span></code></pre></div>
<p><strong>使用场景：</strong></p>
<ul>
<li>✅ 实时通知推送（如日志流、系统监控）</li>
<li>✅ 服务器状态更新（如 MCP 工具列表变更）</li>
<li>✅ 进度反馈（如长时间任务的进度条）</li>
<li>✅ 需要自动重连的场景（浏览器原生支持）</li>
</ul>
<p><strong>优点：</strong></p>
<ol type="1">
<li><strong>自动重连</strong>：断线后浏览器会自动重新连接</li>
<li><strong>轻量级</strong>：比 WebSocket 简单，适合单向推送</li>
<li><strong>跨域友好</strong>：支持 CORS</li>
<li><strong>事件 ID 机制</strong>：支持断线续传（Last-Event-ID）</li>
</ol>
<p><strong>缺点：</strong></p>
<ol type="1">
<li><strong>单向通信</strong>：只能服务器 → 客户端，客户端发请求需要额外
HTTP</li>
<li><strong>浏览器限制</strong>：有些浏览器对同域 SSE 连接数有限制（如 6
个）</li>
<li><strong>代理问题</strong>：某些企业代理会缓冲 SSE 流</li>
<li><strong>不适合双向交互</strong>：如果需要频繁发送请求，不如用
WebSocket</li>
</ol>
<p><strong>典型实现：</strong></p>
<ul>
<li>ChatGPT 的流式输出（早期使用 SSE）</li>
<li>GitHub Actions 的日志流</li>
<li>股票行情推送</li>
</ul>

<h3 id="3-http-request-response">3. HTTP (Request-Response)</h3>
<p><strong>协议本质：</strong></p>
<ul>
<li>经典的 <strong>请求-响应</strong> 模式</li>
<li>每次交互都是独立的 HTTP POST 请求</li>
<li>无状态、可缓存、易于水平扩展</li>
</ul>
<p><strong>工作流程：</strong></p>
<pre class="mermaid"><code>sequenceDiagram
Client-&gt;&gt;MCP Server (HTTP API): POST /mcp, {&quot;method&quot;:&quot;initialize&quot;}
MCP Server (HTTP API)--&gt;&gt;Client:200 OK, {&quot;result&quot;:{...}}
Client-&gt;&gt;MCP Server (HTTP API): POST /mcp, {&quot;method&quot;:&quot;tools/list&quot;}
MCP Server (HTTP API)--&gt;&gt;Client:200 OK, {&quot;result&quot;:{&quot;tools&quot;:[...]}}</code></pre>
<p><strong>使用场景：</strong></p>
<ul>
<li>✅ 云端 API 服务（如天气 API、翻译 API）</li>
<li>✅ 跨机器、跨网络调用</li>
<li>✅ 多客户端共享同一服务（如数据库、知识库）</li>
<li>✅ 需要负载均衡、CDN、缓存等基础设施</li>
</ul>
<p><strong>优点：</strong></p>
<ol type="1">
<li><strong>生态成熟</strong>：HTTP
工具链完善（nginx、HAProxy、CDN）</li>
<li><strong>易于部署</strong>：标准 Web 服务，容器化、K8s 友好</li>
<li><strong>无状态</strong>：每个请求独立，易于水平扩展</li>
<li><strong>易于调试</strong>：curl、Postman、浏览器都能测试</li>
<li><strong>防火墙友好</strong>：几乎所有网络环境都允许 HTTP</li>
</ol>
<p><strong>缺点：</strong></p>
<ol type="1">
<li><strong>连接开销大</strong>：每次请求都需要 TCP 三次握手、TLS
握手</li>
<li><strong>延迟高</strong>：相比本地 IPC，网络延迟 10-100ms</li>
<li><strong>无法主动推送</strong>：服务器无法主动通知客户端（需要轮询或
webhook）</li>
<li><strong>不适合高频调用</strong>：每秒上千次请求会很低效</li>
</ol>
<p><strong>典型实现：</strong></p>
<ul>
<li>RESTful API（如 GitHub API、Stripe API）</li>
<li>微服务架构（Spring Cloud、gRPC Gateway）</li>
<li>Serverless Functions（AWS Lambda HTTP endpoint）</li>
</ul>
<hr />
<h3 id="4-streamable-http">4. Streamable HTTP</h3>
<p><strong>协议本质：</strong></p>
<ul>
<li>HTTP + SSE 的 <strong>混合模式</strong></li>
<li>请求用 HTTP POST，响应可以是流式（SSE）或普通 JSON</li>
<li>通过 <code>mcp-session-id</code> header 维持会话状态</li>
</ul>
<p><strong>工作流程：</strong></p>
<pre class="mermaid"><code>sequenceDiagram
Client-&gt;&gt;MCP Server: POST /mcp (Accept: json+sse), {&quot;method&quot;:&quot;initialize&quot;} 
MCP Server--&gt;&gt;Client: 200 OK, Set-Header: mcp-session-id, {&quot;result&quot;:{...}}
Client-&gt;&gt;MCP Server: POST /mcp, Header: mcp-session-id, {&quot;method&quot;:&quot;tools/call&quot;}
MCP Server--&gt;&gt;Client: 200 OK (streaming), event: progress, data: {&quot;percent&quot;:50}, event: result, data: {&quot;output&quot;:&quot;...&quot;}</code></pre>
<p><strong>使用场景：</strong></p>
<ul>
<li>✅ 长时间任务（如代码生成、视频转码）</li>
<li>✅ 需要进度反馈（如文件上传、批量处理）</li>
<li>✅ 大文件传输（分块流式发送）</li>
<li>✅ 有状态的会话管理（如多轮对话）</li>
</ul>
<p><strong>优点：</strong></p>
<ol type="1">
<li><strong>双向能力</strong>：请求用 HTTP，响应可以流式</li>
<li><strong>会话管理</strong>：通过 session ID 维护上下文</li>
<li><strong>渐进式响应</strong>：适合生成式 AI 的流式输出</li>
<li><strong>兼容性好</strong>：基于标准 HTTP/SSE，无需特殊协议</li>
</ol>
<p><strong>缺点：</strong></p>
<ol type="1">
<li><strong>复杂度高</strong>：需要服务端管理 session 状态</li>
<li><strong>内存开销</strong>：服务端需要存储会话信息</li>
<li><strong>不如 WebSocket 实时</strong>：仍然是 HTTP
请求-响应，有延迟</li>
<li><strong>支持度低</strong>：较新的协议，工具链不成熟</li>
</ol>
<p><strong>典型实现：</strong></p>
<ul>
<li>OpenAI Streaming API（GPT-4 的流式输出）</li>
<li>Anthropic Claude API（支持 streaming）</li>
<li>AWS Bedrock 的流式推理</li>
</ul>

<h2 id="三各大-agent-的协议选择策略">三、各大 Agent 的协议选择策略</h2>
<h3 id="1-claude-desktop">1. Claude Desktop</h3>
<ul>
<li><p><strong>主要协议</strong>：stdio</p></li>
<li><p><strong>设计哲学</strong>：本地优先、隐私保护</p></li>
<li><p><strong>原因</strong>：</p>
<ul>
<li>用户数据不离开本地</li>
<li>工具生态以 npm
包为主（<code>npx @modelcontextprotocol/server-*</code>）</li>
<li>简化部署（无需配置网络）</li>
</ul></li>
</ul>
<h3 id="2-google-gemini-cli">2. Google Gemini CLI</h3>
<ul>
<li><p><strong>支持协议</strong>：stdio, sse, http</p></li>
<li><p><strong>设计哲学</strong>：混合架构（本地 + 云端）</p></li>
<li><p><strong>原因</strong>：</p>
<ul>
<li>stdio：本地工具（文件系统、代码分析）</li>
<li>http：调用 Google Cloud API（知识图谱、企业数据）</li>
<li>sse：实时接收 Gemini 模型的流式输出</li>
</ul></li>
</ul>
<h3 id="3-企业级-agent如-iflow">3. 企业级 Agent（如 iFlow）</h3>

<li><p><strong>原因</strong>：</p>
<ul>
<li>stdio：开发环境的本地工具</li>
<li>http：与企业内部系统集成（ERP、CRM、数据库）</li>
<li>sse：实时监控、告警推送</li>
<li>streamable_http：长时间分析任务（如财务报表生成）</li>
</ul></li>
</ul>

<ul>
<li>Pipe

<li>兼容性好（所有语言都支持按行读取）</li>
</ul>

<h3 id="5-现状统计"><strong>5. 现状统计</strong></h3>
<p>根据 MCP 社区调研：</p>
<ul>
<li><strong>Stdio</strong>: 占 85% 的 MCP 工具（如 npm
上的 <code>@modelcontextprotocol/server-*</code>）</li>
<li><strong>HTTP</strong>: 占 12%（主要是云端 API 包装）</li>
<li><strong>SSE/WebSocket</strong>: 占 3%（实验性）</li>
</ul>
<p><strong>为什么 Stdio 占绝对主导？</strong></p>
<ol type="1">
<li><strong>MCP 最初设计就是本地优先</strong>（Claude Desktop
的设计哲学）</li>
<li><strong>npm/pypi 生态成熟</strong>（工具分发以 CLI 为主）</li>
<li><strong>开发者习惯</strong>（命令行工具比 Web 服务更直观）</li>
<li><strong>安全性要求</strong>（AI Agent
访问本地文件，不能暴露端口）</li>
</ol>
<h2 id="八未来协议发展趋势">八、未来协议发展趋势</h2>
<h3 id="1-websocket-的必然性">1. <strong>WebSocket
的必然性</strong></h3>
<p><strong>为什么会普及：</strong></p>
<ul>
<li>AI Agent 越来越需要 <strong>双向实时交互</strong></li>
<li>多轮对话、工具链调用需要 <strong>低延迟</strong></li>
<li>WebSocket 是目前最成熟的双向协议</li>
</ul>
<p><strong>预测：</strong></p>
<ul>
<li>2025 年底，主流 Agent 会支持 WebSocket</li>
<li>MCP 2.0 可能会正式加入 WebSocket 作为标准传输层</li>
</ul>
<h3 id="2-grpc-在企业场景的崛起">2. <strong>gRPC
在企业场景的崛起</strong></h3>
<p><strong>为什么适合 Agent：</strong></p>
<ul>
<li><strong>性能极高</strong>：Protocol Buffers 比 JSON 快 3-10 倍</li>
<li><strong>强类型</strong>：<code>.proto</code> 定义避免 API
不一致</li>
<li><strong>流式支持</strong>：天然支持双向流</li>
</ul>
<p><strong>应用场景：</strong></p>
<ul>
<li>Agent 之间的内部通信（多 Agent 协作）</li>
<li>高性能场景（如实时翻译、语音识别）</li>
<li>微服务架构（K8s 环境）</li>
</ul>
<h3 id="3-边缘计算与-p2p-mcp">3. <strong>边缘计算与 P2P
MCP</strong></h3>
<p><strong>趋势：</strong></p>
<ul>
<li>隐私意识提升 → 数据不愿意上云</li>
<li>边缘设备算力提升（如 Apple M4、NVIDIA Jetson）</li>
<li>P2P 技术成熟（libp2p、WebRTC）</li>
</ul>
<p><strong>愿景：</strong></p>
<ul>
<li>Agent 直接在本地设备之间通信</li>
<li>去中心化的工具市场（类似 IPFS）</li>
<li>无需服务器的 Agent 协作网络</li>
</ul>
<h3
id="4-协议自适应protocol-negotiation">4. <strong>协议自适应（Protocol
Negotiation）</strong></h3>
<p><strong>核心思想：</strong></p>
<ul>
<li>客户端和服务器协商最优协议（类似 HTTP/2 ALPN）</li>
<li>根据网络状况动态切换</li>
<li>自动降级策略</li>
</ul>
<p><strong>示例流程：</strong></p>
<pre class="mermaid"><code>sequenceDiagram
Client-&gt;&gt;Server: Capabilities, [&quot;stdio&quot;,&quot;ws&quot;,&quot;http&quot;]
Server--&gt;&gt;Client: Preferred: ws
Client-&gt;&gt;Server: WS Connection
Client--&gt;&gt;Server:  Fallback: HTTP  (如果 WS 超时)</code></pre>

<h2 id="九我的个人观点">九、我的个人观点</h2>
<h3 id="1-对协议演进的看法">1. 对协议演进的看法</h3>
<ol type="1">
<li><p><strong>stdio 永远不会消失</strong></p>
<ul>
<li>本地工具、命令行场景无可替代</li>
<li>开发者永远需要简单、快速的本地调用</li>
<li>类比：即使有了 HTTP，shell pipe 仍然是最高效的</li>
</ul></li>
<li><p><strong>HTTP 是过渡方案</strong></p>
<ul>
<li>当前主流是因为生态成熟</li>
<li>但对于实时 Agent 交互，延迟太高</li>
<li>未来会被 WebSocket/gRPC 替代（在需要实时性的场景）</li>
</ul></li>
<li><p><strong>SSE 很尴尬</strong></p>
<ul>
<li>介于 HTTP 和 WebSocket 之间</li>
<li>只适合"服务器单向推送"这个狭窄场景</li>
<li>未来可能被 WebSocket 完全取代</li>
</ul></li>
<li><p><strong>Streamable HTTP 是 AI 时代的产物</strong></p>
<ul>
<li>专为生成式 AI 设计（流式输出）</li>
<li>但实现复杂，采用率低</li>
<li>可能会被 WebSocket + Streaming 取代</li>
</ul></li>
</ol>

<li><strong>向后兼容</strong>：新协议不影响旧工具</li>
</ul>

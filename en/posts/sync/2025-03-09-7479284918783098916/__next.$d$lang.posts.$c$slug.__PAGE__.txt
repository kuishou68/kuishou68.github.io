3:T142d4,<h2 id="1-promise-a规范的基本概念">1. Promise A+规范的基本概念</h2>
<p>Promise是一套专门处理异步场景的规范，它能有效的避免回调地狱的产生，使异步代码更加清晰、简洁、统一</p>
<p>这套规范最早诞生于前端社区，规范名称为<a
href="https://promisesaplus.com/">Promise A+</a></p>
<p>该规范出现后，立即得到了很多开发者的响应</p>
<p>Promise A+ 规定：</p>
<ol type="1">
<li><p>所有的异步场景，都可以看作是一个异步任务，每个异步任务，在JS中应该表现为一个<strong>对象</strong>，该对象称之为<strong>Promise对象</strong>，也叫做任务对象</p></li>
<li><p>每个任务对象，都应该有两个阶段、三个状态</p>
<p>根据常理，它们之间存在以下逻辑：</p>
<ul>
<li>任务总是从未决阶段变到已决阶段，无法逆行</li>
<li>任务总是从挂起状态变到完成或失败状态，无法逆行</li>
<li>时间不能倒流，历史不可改写，任务一旦完成或失败，状态就固定下来，永远无法改变
<img
src="https://p3-xtjj-sign.byteimg.com/tos-cn-i-73owjymdk6/fbf52cf4ee3349cfa0b5630b399a5681~tplv-73owjymdk6-jj-mark-v1:0:0:0:0:5o6Y6YeR5oqA5pyv56S-5Yy6IEAgQ29jb29uQnJlYWs=:q75.awebp?rk3s=f64ab15b&amp;x-expires=1772942507&amp;x-signature=wuGrgTGcdJZQDkI3BUU05tRi1QM%3D"
alt="image.png" /></li>
</ul></li>
<li><p><code>挂起-&gt;完成</code>，称之为<code>resolve</code>；<code>挂起-&gt;失败</code>称之为<code>reject</code>。任务完成时，可能有一个相关数据；任务失败时，可能有一个失败原因。</p></li>
<li><p>可以针对任务进行后续处理，针对完成状态的后续处理称之为onFulfilled，针对失败的后续处理称之为onRejected</p></li>
</ol>

<h2 id="4promise链式调用">4.Promise链式调用</h2>
<ol type="1">
<li><p>then方法必须会返回一个新的promise, 可以理解为
<strong>后续处理也是一个任务</strong></p></li>
<li><p>新任务的状态取决于后续处理：</p>
<ul>
<li>若没有相关的后续处理，新任务的状态和前任务一致，数据为前任务的状态。</li>
<li>若有后续处理但未执行，新任务挂起。</li>
<li>若后续处理执行了，则根据后续处理的情况确定新任务的状态。
<ul>
<li>后续处理执行无错，新任务的状态未完成，数据为后续处理的返回值。</li>
<li>后续处理执行有错，新任务的状态为失败，数据为异常数据。</li>
<li>后续执行后返回时一个任务对象，新任务的状态和数据与该任务对象一致。</li>
</ul></li>
</ul></li>
</ol>

<p>这与 JavaScript 的 <strong>微任务调度机制</strong>
有关。让我们一步步拆解：</p>
<h4 id="1-promise-的链式调用与微任务队列">1. Promise
的链式调用与微任务队列</h4>
<ul>
<li>在 JavaScript 中，Promise 的 <code>.then</code>、<code>.catch</code>
和 <code>.finally</code> 都会返回一个新的 Promise 对象。</li>
<li>当一个 Promise 完成（resolved 或 rejected）时，它注册的回调（如
<code>.then</code> 或 <code>.finally</code> 中的函数）会被加入
<strong>微任务队列</strong>。</li>
<li>微任务队列是先进先出（FIFO）的，任务会按照加入队列的顺序依次执行。</li>
</ul>
<h4 id="2-同步代码的执行">2. 同步代码的执行</h4>
<p>当代码运行时，同步部分会立即执行：</p>
<ul>
<li><strong>第一个 Promise</strong>：<code>Promise.resolve('1')</code>
创建一个立即 resolved 的 Promise，值为 <code>'1'</code>。它的
<code>.then</code>
回调（<code>console.log(res)</code>）被加入微任务队列。</li>
<li><strong>第二个 Promise</strong>：<code>Promise.resolve('2')</code>
也创建一个立即 resolved 的 Promise，值为 <code>'2'</code>。它的
<code>.finally</code>
回调（<code>console.log('finally2')</code>）也被加入微任务队列。</li>
</ul>
<p>同步代码执行完后，微任务队列的初始状态是：</p>
<ul>
<li>队列：[<code>第一个 Promise 的 .then</code>,
<code>第二个 Promise 的 .finally</code>]</li>
</ul>
<h4 id="3-微任务的逐步执行">3. 微任务的逐步执行</h4>
<p>JavaScript
的事件循环会在同步代码执行完后，清空微任务队列。让我们看看具体过程：</p>
<ul>
<li><p><strong>执行第一个任务</strong>：</p>
<ul>
<li>从队列中取出 <code>第一个 Promise 的 .then</code>，执行
<code>console.log('1')</code>，打印 <code>'1'</code>。</li>
<li><code>.then</code> 执行后，返回一个新的 Promise（默认
resolved），并将
<code>.finally(() =&gt; { console.log('finally') })</code>
加入微任务队列。</li>
<li>此时队列变为：[<code>第二个 Promise 的 .finally</code>,
<code>第一个 Promise 的 .finally</code>]</li>
</ul></li>
<li><p><strong>执行第二个任务</strong>：</p>
<ul>
<li>队列中下一个任务是 <code>第二个 Promise 的 .finally</code>，执行
<code>console.log('finally2')</code>，打印
<code>'finally2'</code>。</li>
<li><code>.finally</code> 执行后，返回一个新的 Promise（值为原始的
<code>'2'</code>），并将
<code>.then(res =&gt; { console.log('finally2后面的then函数', res) })</code>
加入微任务队列。</li>
<li>队列变为：[<code>第一个 Promise 的 .finally</code>,
<code>第二个 Promise 的 .then</code>]</li>
</ul></li>
<li><p><strong>执行第三个任务</strong>：</p>
<ul>
<li>执行 <code>第一个 Promise 的 .finally</code>，打印
<code>'finally'</code>。</li>
<li>队列变为：[<code>第二个 Promise 的 .then</code>]</li>
</ul></li>
<li><p><strong>执行第四个任务</strong>：</p>
<ul>
<li>执行 <code>第二个 Promise 的 .then</code>，打印
<code>'finally2后面的then函数 2'</code>。</li>
<li>队列清空，执行结束。</li>
</ul></li>
</ul>
<h4 id="4-关键点解答">4. 关键点解答</h4>
<p>为什么在打印 <code>'1'</code> 后没有立即执行
<code>'finally'</code>？原因在于：</p>
<ul>
<li>当第一个 Promise 的 <code>.then</code> 执行时，它的
<code>.finally</code>
回调并不是立即执行，而是被加入微任务队列的末尾。</li>
<li>此时，第二个 Promise 的 <code>.finally</code>
已经在队列中（因为它在同步代码执行时就加入了），而且排在
<code>第一个 Promise 的 .finally</code> 前面。</li>
<li>微任务队列严格遵循先进先出的原则，因此在执行完
<code>第一个 Promise 的 .then</code> 后，会先执行队列中已有的
<code>第二个 Promise 的 .finally</code>（打印
<code>'finally2'</code>），然后才轮到
<code>第一个 Promise 的 .finally</code>（打印
<code>'finally'</code>）。</li>
</ul>
<p>为了更直观地展示，以下是队列的演变过程：</p>
<ol type="1">
<li><p><strong>初始状态</strong>（同步代码后）：</p>
<ul>
<li>队列：[<code>第一个 Promise 的 .then</code>,
<code>第二个 Promise 的 .finally</code>]</li>
</ul></li>
<li><p><strong>执行 .then 后</strong>（打印 <code>'1'</code>）：</p>
<ul>
<li>队列：[<code>第二个 Promise 的 .finally</code>,
<code>第一个 Promise 的 .finally</code>]</li>
</ul></li>
<li><p><strong>执行第二个 .finally 后</strong>（打印
<code>'finally2'</code>）：</p>
<ul>
<li>队列：[<code>第一个 Promise 的 .finally</code>,
<code>第二个 Promise 的 .then</code>]</li>
</ul></li>
<li><p><strong>执行第一个 .finally 后</strong>（打印
<code>'finally'</code>）：</p>
<ul>
<li>队列：[<code>第二个 Promise 的 .then</code>]</li>
</ul></li>
<li><p><strong>执行第二个 .then 后</strong>（打印
<code>'finally2后面的then函数 2'</code>）：</p>
<ul>
<li>队列：空</li>
</ul></li>
</ol>
<p>在执行第一个 Promise 的 <code>.then</code> 并打印 <code>'1'</code>
后，它的 <code>.finally</code> 没有立即执行，是因为
<code>.finally</code> 的回调是在 <code>.then</code>
执行后才加入微任务队列的。而第二个 Promise 的 <code>.finally</code>
在同步代码阶段就已加入队列，排在前面。因此，微任务调度机制决定先执行
<code>finally2</code>，然后才执行第一个 Promise 的
<code>finally</code>。这种顺序是由微任务队列的 FIFO
特性决定的，确保了异步任务的执行有条不紊。</p>
<h2 id="总结">总结</h2>
<ul>
<li><strong>优先级</strong>：同步代码 &gt; 微任务 &gt;
宏任务；微任务队列必须清空后才会处理下一个宏任务。</li>
<li><strong>状态不可逆</strong>：一旦从 Pending
变为其他状态，不能再改变；</li>
<li>js代码<strong>永远不会卡住</strong>，后面有代码就会继续执行后面的；</li>
<li>任何一个 <code>.then()</code> 抛出错误，会直接跳到最近的 <code>.catch()</code>；</li>
<li>then里面如果传的不是函数，直接忽略这段代码，因为promise的then发现你传递的不是一个promise，也就是没有注册回调函数，那它的结果和状态和上一个promise是一致的，相当于 <code>.then(null)</code>;</li>
<li><code>.finally()</code>方法的回调函数不接受任何的参数，不管Promise对象最后的状态如何都会执行；</li>
<li>微任务队列是<strong>先进先出</strong>（FIFO）的，任务会按照加入队列的顺序依次执行；</li>
<li>.then、.catch 和 .finally 会返回一个新的 Promise
对象，<strong>事件循环</strong>会在同步代码执行完后，清空微任务队列。</li>
</ul>

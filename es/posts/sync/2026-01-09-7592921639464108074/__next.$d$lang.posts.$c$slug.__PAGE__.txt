<h2 id="什么是-codex">什么是 Codex？</h2>
<p>想象一下，你有一个超级聪明的编程助手，它不仅能理解你的需求，还能直接帮你写代码、运行命令、修改文件。这就是
<strong>Codex</strong> —— OpenAI 推出的本地 AI 编程代理。</p>
<h3 id="用大白话解释">用大白话解释</h3>
<p>如果把编程比作做菜：</p>
<ul>
<li><strong>普通的 AI 助手</strong>（如
ChatGPT）就像一本菜谱，告诉你怎么做，但你需要自己动手</li>
<li><strong>Codex</strong>
就像一个真正的厨师，不仅告诉你怎么做，还能直接帮你把菜做出来</li>
</ul>
<h3 id="核心能力">核心能力</h3>
<ul>
<li>Codex 能做什么？
<ul>
<li>理解代码：阅读和分析你的项目代码</li>
<li>编写代码：自动编写新功能或修复 bug</li>
<li>执行命令：运行测试、构建项目、git 操作等</li>
<li>修改文件：直接编辑、创建、删除文件</li>
<li>搜索信息：在代码库中搜索，甚至进行网络搜索</li>
<li>调用外部工具：通过 MCP 协议连接各种外部服务</li>
</ul></li>
</ul>
<h2 id="整体架构概览">整体架构概览</h2>

的组织方式，主要由 Rust 和 TypeScript 两种语言构建。</p>

<h2 id="底层协议与通信机制">底层协议与通信机制</h2>
<h3 id="提交队列事件队列sqeq模式">提交队列/事件队列（SQ/EQ）模式</h3>
<p>Codex 内部使用了一种高效的异步通信模式：</p>
<pre class="mermaid"><code>
sequenceDiagram
    participant User as 用户/TUI
    participant SQ as 提交队列 (SQ)
    participant Engine as Codex 核心引擎
    participant LLM as 大语言模型
    participant Tools as 工具系统
    participant EQ as 事件队列 (EQ)

    User-&gt;&gt;SQ: 提交用户输入
    Note over SQ: Op::UserTurn

    SQ-&gt;&gt;Engine: 异步接收提交
    Engine-&gt;&gt;LLM: 调用模型 API

    LLM--&gt;&gt;Engine: 流式响应
    Engine-&gt;&gt;EQ: 发送 AgentMessage 事件
    EQ--&gt;&gt;User: 实时显示 AI 回复

    LLM--&gt;&gt;Engine: 工具调用请求
    Engine-&gt;&gt;Tools: 执行工具
    Engine-&gt;&gt;EQ: 发送 ExecApprovalRequest
    EQ--&gt;&gt;User: 请求用户批准

    User-&gt;&gt;SQ: 提交批准决定
    SQ-&gt;&gt;Engine: 接收批准

    Tools--&gt;&gt;Engine: 工具执行结果
    Engine-&gt;&gt;EQ: 发送 ExecCommandOutput
    EQ--&gt;&gt;User: 显示执行结果

    Engine-&gt;&gt;LLM: 继续推理
    LLM--&gt;&gt;Engine: 最终响应
    Engine-&gt;&gt;EQ: 发送 TurnCompleted
    EQ--&gt;&gt;User: 显示完成状态
</code></pre>
<p><strong>用大白话解释</strong>：</p>
<p>想象一家餐厅：</p>
<ul>
<li><strong>提交队列（SQ）</strong> 就像顾客下单的窗口</li>
<li><strong>Codex 核心引擎</strong> 就像厨房</li>
<li><strong>事件队列（EQ）</strong> 就像出餐窗口</li>
</ul>
<p>顾客下单后不需要等在窗口，可以去座位上等。厨房做好菜后，通过出餐窗口通知顾客取餐。</p>

<h3 id="问题上下文窗口的限制">问题：上下文窗口的限制</h3>
<p>大语言模型有一个"上下文窗口"限制，就像人的工作记忆一样有限。如果对话太长，模型就会"记不住"前面的内容。</p>
<pre><code>┌────────────────────────────────────────────────────────────┐
│                    上下文窗口示意图                          │
├────────────────────────────────────────────────────────────┤
│  [系统提示] [用户1] [AI1] [用户2] [AI2] ... [用户N] [AI N]  │
│  ←─────────────── 有限的容量 ──────────────────→           │
│                                                            │
│  超出容量时：旧的内容会被&quot;遗忘&quot;                              │
└────────────────────────────────────────────────────────────┘</code></pre>
<h3 id="解决方案智能上下文管理">解决方案：智能上下文管理</h3>
<p>Codex 采用多种策略来最大化利用有限的上下文：</p>
<pre class="mermaid"><code>
flowchart TD
    Start[&quot;开始新的对话轮次&quot;] --&gt; Check{&quot;检查上下文大小&quot;}

    Check --&gt;|&quot;&lt; 80% 窗口&quot;| Normal[&quot;正常添加内容&quot;]
    Check --&gt;|&quot;&gt;= 80% 窗口&quot;| Strategy[&quot;应用优化策略&quot;]

    Strategy --&gt; S1[&quot;策略1: Token 估算&quot;]
    Strategy --&gt; S2[&quot;策略2: 智能截断&quot;]
    Strategy --&gt; S3[&quot;策略3: 对话压缩&quot;]
    Strategy --&gt; S4[&quot;策略4: 环境差异&quot;]

    S1 --&gt; |&quot;每4字节≈1 token&quot;| Estimate[&quot;快速估算大小&quot;]
    S2 --&gt; |&quot;保留头尾&quot;| Truncate[&quot;删除中间内容&quot;]

<h3 id="策略-2智能截断">策略 2：智能截断</h3>
<p>对于过长的输出，保留头尾，删除中间：</p>
<pre><code>原始输出（10000 字符）：
┌─────────────────────────────────────────────────────────────┐
│ [头部 1000 字符] [...中间内容...] [尾部 1000 字符]          │
└─────────────────────────────────────────────────────────────┘

截断后（2500 字符）：
┌─────────────────────────────────────────────────────────────┐
│ [头部 1000 字符] ... [省略 7500 字符] ... [尾部 1000 字符]  │
└─────────────────────────────────────────────────────────────┘</code></pre>
<h3 id="策略-3对话压缩compaction">策略 3：对话压缩（Compaction）</h3>
<p>当上下文接近满时，Codex 会请求 AI 对之前的对话进行"摘要"：</p>
<pre><code>压缩前：
用户: 帮我看看为什么测试失败
AI: 好的，让我运行测试... 测试结果是... 问题在于...
用户: 修复它
AI: 我修改了文件... 现在测试通过了
用户: 还有其他问题吗？
...（很多轮对话）

压缩后：
[摘要] 用户请求修复测试失败问题，经过分析发现是类型错误，
已修改 src/utils.ts 第 42 行，测试现已通过。</code></pre>

<pre><code>用户: 帮我清理项目

<p>沙箱就是为了防止这种灾难性的操作。</p>

<h2 id="执行策略智能的权限管理">执行策略：智能的权限管理</h2>
<h3 id="问题如何判断命令是否安全">问题：如何判断命令是否安全？</h3>
<p>不是所有命令都同样危险：</p>
<ul>
<li><code>ls</code> 完全安全</li>
<li><code>npm test</code> 通常安全</li>
<li><code>rm -rf /</code> 绝对危险</li>
<li><code>git push</code> 需要确认</li>
</ul>
<h3 id="解决方案执行策略引擎">解决方案：执行策略引擎</h3>
<p>Codex 使用基于规则的执行策略系统：</p>
<pre class="mermaid"><code>
flowchart TD
    Start[&quot;命令: git push origin main&quot;] --&gt; Parse[&quot;解析命令 tokens&quot;]
    Parse --&gt; Lookup[&quot;查找匹配规则&quot;]

    subgraph Rules[&quot;规则库&quot;]
        R1[&quot;prefix_rule([&#39;ls&#39;], &#39;allow&#39;)&quot;]
        R2[&quot;prefix_rule([&#39;git&#39;, &#39;push&#39;], &#39;prompt&#39;)&quot;]
        R3[&quot;prefix_rule([&#39;rm&#39;, &#39;-rf&#39;], &#39;forbidden&#39;)&quot;]
    end

    Lookup --&gt; R1
    Lookup --&gt; R2
    Lookup --&gt; R3

    R1 --&gt; Match{&quot;匹配检查&quot;}
    R2 --&gt; Match
    R3 --&gt; Match

    Match --&gt;|&quot;匹配 R2&quot;| Collect[&quot;收集匹配结果&quot;]
    Match --&gt;|&quot;无匹配&quot;| Heuristics[&quot;启发式检查&quot;]

    Heuristics --&gt;|&quot;安全命令&quot;| AllowH[&quot;Decision: Allow&quot;]
    Heuristics --&gt;|&quot;危险命令&quot;| PromptH[&quot;Decision: Prompt&quot;]

    Collect --&gt; MaxDecision[&quot;选择最严格决策&quot;]

    MaxDecision --&gt; FinalAllow[&quot;Allow: 直接执行&quot;]
    MaxDecision --&gt; FinalPrompt[&quot;Prompt: 请求用户批准&quot;]
    MaxDecision --&gt; FinalForbid[&quot;Forbidden: 拒绝执行&quot;]

    AllowH --&gt; FinalAllow
    PromptH --&gt; FinalPrompt

    FinalPrompt --&gt; UserChoice{&quot;用户选择&quot;}
    UserChoice --&gt;|&quot;批准&quot;| Execute[&quot;执行命令&quot;]
    UserChoice --&gt;|&quot;白名单&quot;| Whitelist[&quot;添加 allow 规则&quot;]
    UserChoice --&gt;|&quot;拒绝&quot;| Reject[&quot;取消执行&quot;]

    Whitelist --&gt; Execute

    style Start fill:#2196F3,color:#fff
    style FinalAllow fill:#4CAF50,color:#fff
    style FinalPrompt fill:#FF9800,color:#fff
    style FinalForbid fill:#F44336,color:#fff
    style Rules fill:#E8F5E9,stroke:#388E3C
</code></pre>
